# 骰子传说开发计划

## 目标概述

本计划合并「Dice Legend 项目优化计划」「Cloudflare 后端代理细化方案」「用户自填 API Key 的 Cloudflare 优化方案」。项目目标是将现有 Vite + React + Three.js 骰子抽卡游戏优化为更安全、可部署到 Cloudflare、支持用户自填第三方 API Key 的网页游戏。

当前主策略：

- 前端保留核心 3D 投骰、奖励选择、角色卡和召唤体验。
- 第三方 API 改为 BYOK 模式，由用户在网页中填写自己的 OpenRouter、RunningHub、MiniMax Key。
- Cloudflare Worker 作为同域 `/api/*` 代理，只做安全转发、请求校验和错误规整，不保存平台统一密钥。
- 未配置某个 API Key 时，对应能力自动缺失，主流程不中断。

## 架构计划

### 前端架构

- 新增 API 设置面板，集中管理 OpenRouter、RunningHub、MiniMax Key 和 OpenRouter 模型名。
- 新增本地 Key 存储层，负责从 `localStorage` 读取、保存、清除用户配置。
- 新增能力检测层，根据 Key 是否存在判断可用能力：
  - OpenRouter：角色文案、立绘提示词、动态提示词、语音台词。
  - RunningHub：角色立绘、角色动态化视频。
  - MiniMax：角色语音。
- 新增本域 API Client，所有第三方调用统一走 `/api/*`，避免组件直接调用第三方 URL。
- 后续继续拆分 `App.tsx` 和 `CharacterCard.tsx`，把长流程状态机、RunningHub 轮询、OpenRouter prompt 构造进一步迁出组件。

### Cloudflare 架构

- 采用单 Worker 部署：
  - Worker 托管 Vite `dist` 静态资源。
  - `/api/*` 由 Worker 优先处理。
  - 其他路径走静态资源和 SPA 回退。
- `wrangler.jsonc` 使用：
  - `assets.directory: "./dist"`
  - `assets.not_found_handling: "single-page-application"`
  - `assets.run_worker_first: ["/api/*"]`
- Worker 不保存第三方 API Key；Key 由用户浏览器在请求时通过专用请求头传入。

## API 代理计划

当前白名单接口：

- `POST /api/openrouter/chat`
- `POST /api/runninghub/run`
- `POST /api/runninghub/outputs`
- `POST /api/minimax/voice-design`
- `POST /api/minimax/t2a`

统一响应：

- 成功：`{ "ok": true, "data": ... }`
- 失败：`{ "ok": false, "error": { "code": "...", "message": "..." } }`

安全约束：

- 只允许白名单 endpoint，不提供任意 URL 代理。
- 只接受 `POST` 和 `application/json`。
- 限制请求体大小。
- 校验同源 `Origin`，本地允许 `localhost` 和 `127.0.0.1`。
- 不打印 `X-User-Api-Key`、`Authorization`、RunningHub `apiKey` 或大体积音频字段。
- 不将 Key 写入 URL、构建产物或仓库。

## 功能降级计划

未配置 Key 时的表现：

- 无 OpenRouter：
  - 角色文案使用本地 `generateFallbackInfo`。
  - 立绘提示词使用本地模板。
  - 动态提示词和语音台词使用本地兜底。
- 无 RunningHub：
  - 契约流程跳过立绘生成。
  - 角色卡显示文字和占位视觉。
  - Live 动态化按钮禁用，并提示未配置 RunningHub Key。
- 无 MiniMax：
  - 跳过语音生成。
  - 角色卡显示「未配置 MiniMax API Key，角色语音未生成」。

## UI 与交互计划

已确定方向为「游戏主界面优先」：

- 主界面品牌显示为「骰子传说」。
- API 设置入口放在左上角主面板中，和风格设置并列。
- API 设置面板展示三项能力状态：文案、立绘/动态、语音。
- OpenRouter 除 Key 外支持配置模型名，默认 `x-ai/grok-4.1-fast`，主要用于文案生成，也会复用到其他 OpenRouter 生成任务。
- 缺失能力用禁用态、提示态和占位视觉表达，而不是让流程直接崩溃。

## 安全与合规计划

- 移除旧的 `VITE_*` 前端环境变量密钥模式。
- 删除 Vite 中 `process.env.GEMINI_API_KEY` 的构建注入。
- `.env.example` 改为说明 BYOK 模式，不再提供前端密钥字段。
- `.gitignore` 忽略 `.env*`、`.dev.vars*`、`.wrangler/`、`worker-configuration.d.ts`。
- 文案和生成提示词改为全年龄幻想战棋风格，避免露骨或性化内容。
- `npm audit fix` 修补当前依赖漏洞，并在验证中保持 0 vulnerabilities。

## 测试与验收计划

每轮关键改动后执行：

- `npm run typecheck`
- `npm run build`
- `npx wrangler deploy --dry-run`
- `npm audit --audit-level=moderate`
- 构建产物中搜索旧环境变量名和敏感字段：
  - `OPENROUTER_API_KEY`
  - `RUNNINGHUB_API_KEY`
  - `MINIMAX_API_KEY`
  - `GEMINI_API_KEY`
  - `VITE_`

手动验收场景：

- 三个 Key 都为空：可以投骰并显示文字角色卡。
- 只填 OpenRouter：文案可生成，立绘/语音缺失态正常。
- 只填 RunningHub：立绘可生成，OpenRouter 相关内容使用本地兜底。
- 只填 MiniMax：语音依赖角色生成流程，缺 OpenRouter 时台词使用本地兜底。
- OpenRouter 模型改成其他模型名后，请求体中的 `model` 使用用户配置值。
- Cloudflare 本地预览访问 `http://localhost:8787`，刷新页面和 `/api/*` 代理均正常。

## 后续优化方向

- 将 `App.tsx` 的契约生成流程拆为 `useContractGeneration` 或 service 编排层。
- 将 `CharacterCard.tsx` 的 Live 动态化逻辑拆为独立 hook/service。
- 对 Three.js 和大组件做动态导入，降低当前构建 chunk 体积。
- 增加端到端测试，覆盖缺 Key 降级与 Worker 代理错误响应。
- 若未来要提供平台默认 Key，可在 Worker 增加可选服务端密钥模式，但必须明确优先级和成本控制。
