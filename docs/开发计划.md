# 骰子传说详细开发计划

记录日期：2026-05-02

## 1. 计划来源与最终决策

本文档合并并细化以下三份计划：

- Dice Legend 项目优化计划
- Cloudflare 后端代理细化方案
- 用户自填 API Key 的 Cloudflare 优化方案

三份计划之间有一次关键策略变化：

- 早期安全方案假设由服务端统一持有 OpenRouter、RunningHub、MiniMax 密钥。
- 后续产品决策改为 BYOK 模式，即用户在前端填写自己的第三方 API Key。
- 因此 Cloudflare Worker 的最终职责不是保存平台密钥，而是作为同域安全代理，负责白名单转发、请求校验、错误规整和避免浏览器 CORS 问题。

最终采用的方案：

- 前端：用户自填 Key，Key 保存在本机浏览器 `localStorage`。
- 后端：Cloudflare Worker 同时托管前端静态资源和 `/api/*` 代理。
- 第三方调用：前端不直连 OpenRouter、RunningHub、MiniMax，一律调用本域 `/api/*`。
- 功能缺失：未配置某项 Key 时，对应能力自动降级，主流程不中断。

## 3. 总体目标

### 3.1 架构目标

- 将第三方 API 调用从 UI 组件中抽出，统一进入 `utils/apiClient.ts`。
- 将用户 API Key 管理从业务组件中抽出，统一进入 `utils/apiKeyStore.ts`。
- 将 Cloudflare Worker 代理独立放在 `src/worker/index.ts`。
- 保留现有核心玩法，不大规模重写 Three.js 物理骰子逻辑。
- 后续继续拆分大组件，逐步降低 `App.tsx` 和 `CharacterCard.tsx` 的维护成本。

### 3.2 安全目标

- 不再将第三方 API Key 写入 Vite 前端环境变量。
- 不再通过 `define` 注入任何第三方密钥。
- 不在 URL 中传递 Key。
- 不在日志中打印 Key、Authorization、RunningHub `apiKey`、音频大字段。
- Worker 只代理白名单接口，不提供任意 URL 代理能力。
- 构建产物中不得出现旧环境变量名或真实密钥。

### 3.3 UI 目标

- 主界面品牌显示为「骰子传说」。
- 左上角提供 API 设置入口。
- API 设置面板展示当前能力状态。
- OpenRouter 除 Key 外支持配置模型名。
- 缺 Key 时显示明确的禁用态、缺失态和占位视觉，而不是抛错打断流程。

### 3.4 部署目标

- 使用 Cloudflare Worker 单体部署：
  - 托管 Vite `dist` 静态资源。
  - `/api/*` 由 Worker 处理。
  - 前端刷新和深路径访问走 SPA 回退。
- 使用 Wrangler 管理本地预览和部署。

## 5. Cloudflare 后端代理详细计划

### 5.1 部署形态

最终部署形态为单 Worker：

- Worker 托管 `dist`。
- Worker 优先处理 `/api/*`。
- 其他请求交给静态资源。
- 未命中静态资源时使用 SPA 回退。

`wrangler.jsonc` 关键配置：

```jsonc
{
  "name": "dice-legend-webgame",
  "main": "./src/worker/index.ts",
  "compatibility_date": "2026-05-02",
  "compatibility_flags": ["nodejs_compat"],
  "observability": {
    "enabled": true
  },
  "assets": {
    "directory": "./dist",
    "binding": "ASSETS",
    "not_found_handling": "single-page-application",
    "run_worker_first": ["/api/*"]
  }
}
```

### 5.2 与早期服务端密钥方案的差异

早期计划：

- 服务端保存 `OPENROUTER_API_KEY`、`RUNNINGHUB_API_KEY`、`MINIMAX_API_KEY`。
- 使用 `wrangler secret put` 上传生产密钥。

最终计划：

- 项目不保存平台统一 Key。
- 用户在前端填写自己的 Key。
- Worker 从请求头 `X-User-Api-Key` 获取用户本次请求使用的 Key。
- Worker 不存储、不落盘、不缓存、不打印用户 Key。

保留的能力：

- 同域代理。
- 白名单 endpoint。
- 请求体白名单化。
- 统一错误响应。
- CORS 和 Origin 约束。
- Cloudflare 单 Worker 部署。

### 5.3 Worker 接口

当前白名单接口：

- `POST /api/openrouter/chat`
  - 上游：`https://openrouter.ai/api/v1/chat/completions`
  - Key 位置：`Authorization: Bearer <用户 OpenRouter Key>`
  - 请求体白名单：`model`、`messages`、`max_tokens`、`temperature`
- `POST /api/runninghub/run`
  - 上游：`https://www.runninghub.cn/task/openapi/ai-app/run`
  - Key 位置：RunningHub 请求体 `apiKey`
  - 请求体白名单：`webappId`、`nodeInfoList`
- `POST /api/runninghub/outputs`
  - 上游：`https://www.runninghub.cn/task/openapi/outputs`
  - Key 位置：RunningHub 请求体 `apiKey`
  - 请求体白名单：`taskId`
- `POST /api/minimax/voice-design`
  - 上游：`https://api.minimaxi.com/v1/voice_design`
  - Key 位置：`Authorization: Bearer <用户 MiniMax Key>`
  - 请求体白名单：`prompt`、`preview_text`、`voice_id`
- `POST /api/minimax/t2a`
  - 上游：`https://api.minimaxi.com/v1/t2a_v2`
  - Key 位置：`Authorization: Bearer <用户 MiniMax Key>`
  - 请求体白名单：`model`、`text`、`stream`、`voice_setting`、`audio_setting`

统一响应格式：

```json
{ "ok": true, "data": {} }
```

```json
{ "ok": false, "error": { "code": "MISSING_API_KEY", "message": "缺少对应的 API Key" } }
```

### 5.4 Worker 安全细节

请求校验：

- 非 `/api/*` 请求直接交给 `env.ASSETS.fetch(request)`。
- `/api/*` 仅允许 `POST`。
- `OPTIONS` 返回预检响应。
- `Content-Type` 必须包含 `application/json`。
- `Content-Length` 超过 64 KB 时拒绝。
- 实际读取 body 后再次检查长度。
- `Origin` 存在时必须与当前 Worker 同源；本地允许 `localhost` 和 `127.0.0.1`。

代理限制：

- endpoint 必须存在于 `targets` 白名单。
- Worker 不接受客户端传来的任意 URL。
- Worker 不透传客户端任意 headers。
- Worker 对每个上游接口单独构造请求体。

日志策略：

- 不打印用户 Key。
- 不打印上游 Authorization。
- 不打印完整音频数据。
- 上游错误时只记录 endpoint、HTTP 状态、耗时。

## 6. 用户自填 API Key 详细计划

### 6.1 配置项

API 设置面板包含：

- OpenRouter API Key
  - 用于角色文案、立绘提示词、动态提示词、语音台词。
- OpenRouter 模型
  - 默认 `x-ai/grok-4.1-fast`。
  - 用户可填写其他 OpenRouter 模型 ID。
  - 主要用于文案生成，也复用于其他 OpenRouter 任务。
- RunningHub API Key
  - 用于角色立绘和动态化视频。
- MiniMax API Key
  - 用于角色语音。

保存策略：

- 保存位置：浏览器 `localStorage`。
- 存储 key：`diceLegend.apiKeys.v1`。
- 清除方式：API 设置面板中的「清除密钥」按钮。
- 旧用户配置兼容：如果本地已有旧结构，会自动补上默认 OpenRouter 模型。

### 6.2 能力检测

`getApiCapabilities` 根据 Key 是否为空生成：

- `openRouter`
- `runningHub`
- `miniMax`

UI 和业务流程都使用能力检测，而不是到调用失败后才处理。

### 6.3 功能降级

无 OpenRouter：

- 角色基础信息生成：
  - 使用 `generateFallbackInfo(result, stylePrompt)`。
- 立绘提示词：
  - 使用 `buildLocalImagePrompt(generatedInfo)`。
- Live 动态提示词：
  - 使用本地稀有度动作模板。
- 语音台词：
  - 使用本地 `buildFallbackVoiceConfig`。

无 RunningHub：

- 角色契约流程：
  - 跳过 RunningHub 立绘生成。
  - 显示文字角色卡和占位视觉。
- 角色卡 Live：
  - 无立绘时禁用。
  - 未配置 RunningHub Key 时禁用。

无 MiniMax：

- 契约流程：
  - 不启动语音生成 Promise。
- 角色卡：
  - 显示语音缺失提示。
  - 语音槽位保持锁定态。

## 13. 风险与注意事项

- BYOK 模式下，Key 必然存在于用户浏览器；本项目的安全边界是避免提交、避免构建注入、避免 URL 泄露、避免日志泄露。
- `localStorage` 是明文存储，适合个人设备；共享设备必须提示用户清除密钥。
- Worker 不能完全防止用户自己的 Key 被其浏览器环境读取；它只负责减少 CORS、日志、URL 和构建泄露风险。
- RunningHub 轮询仍由前端驱动，用户关闭页面会中断前端轮询。
- OpenRouter 模型名完全由用户填写，若模型不可用，上游会返回错误；前端应保持友好错误提示。
- 当前主包体积较大，后续 UI/动效优化前应考虑代码拆分。
- MiniMax 若返回 `invalid api key`，优先判断为用户 Key 与 `platform.minimaxi.com` 平台/API 权限不匹配；当前 Worker 上游域名已按官方文档使用 `api.minimaxi.com`。
- GitHub Actions 已出现 Node.js 20 action 运行时弃用提示，需在后续维护中关注 2026-06-02 后默认 Node.js 24 切换。
